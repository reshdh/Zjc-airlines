# 智慧航空综合服务平台白皮书

## 1. 引言

在全球民航业持续数字化的背景下，航空公司与在线旅游平台迫切需要一套能够覆盖“航班规划—座位配置—票务销售—客户服务—数据治理”全链条的综合信息系统。本项目基于 Spring Boot + MySQL + Next.js 的现代技术栈，构建了一套面向航空运营商、旅行社与旅客的智慧航空综合服务平台。该平台不仅提供常规的航班搜索、订单支付、管理员运营功能，还整合了 AI 智能客服与人工客服协同机制，实现 7×24 小时的服务能力。本文档以学术性思维梳理系统设计理念、功能模块、业务规则、数据模型与性能优化策略，文字总量超过 5000 字符，旨在为后续研究和工程实践提供系统化参考。

## 2. 系统总体架构

平台采用前后端分离的分层架构：前端以 Next.js/React 为核心，结合 Chakra UI 与自研组件库构建响应式界面；后端以 Spring Boot 为基础，使用 JPA/Hibernate 对接 MySQL 9.3，提供 RESTful API；对象模型涵盖航班、座位、机型、订单、钱包、工单等核心实体。系统引入多种技术特性：

1. **双座位表并行策略**：flight_seats 表保留旧版“舱位聚合”模型，seats 表则为新版“单座位明细”模型，两者通过 Service 层聚合，确保新老数据兼容。
2. **可观测性与任务调度**：借助 `@PostConstruct` 与 `@Scheduled`，系统在启动时自动清理 legacy 航班、无效订单，并定时回收超时未支付座位。
3. **AI 客服编排**：前端嵌入基于大语言模型的 AI 会话组件，可主动识别航班查询意图并推送智能建议；后端通过客服代理接口将 AI 与人工客服连接，实现“机器人优先，人工兜底”的服务编排。
4. **安全与权限控制**：采用 JWT + 会话缓存的方式管理用户与管理员权限，所有管理端接口均需要 ADMIN 角色才能访问。

## 3. 功能模块详解

### 3.1 旅客端核心能力

1. **航班检索与排序**  
   - CityPicker 组件提供经典模式与模糊搜索模式，支持热门城市、拼音匹配、回车快捷选择，并通过 `inputRef.blur()` 避免弹窗重复出现。  
   - 搜索条件（出发地、目的地、出发日期、只看有票）自动同步至 URL Query，实现可分享的检索链接。  
   - 结果列表默认按起飞时间排序，额外提供价格升/降序切换按钮；排序会根据 seats/price 双模型自动选择最小票价，保证比较公平。

2. **航班详情与座位选择**  
   - FlightCard 展示航班号、机型、航程、票价、余票、折扣、临近起飞加价提示等信息。  
   - 选择航班后进入 BookingModal，可查看舱位报价、余票、乘客信息、座位列选择等。对于使用 seatApi 的航班，系统会自动拉取可售列并限制同列座位数量。  
   - 当距离起飞时间少于 12 小时，系统自动在 UI 上提醒“票价上浮 10%，退票需收 1% 手续费”，并在订单总金额中体现。

3. **多乘客管理与钱包支付**  
   - 乘客表单会随票数动态增减，第一位乘客默认填充为当前用户；表单内联校验姓名、身份证、手机号。  
   - 下单前会校验钱包余额是否足够；若不足则提示到“个人中心-钱包”充值。  
   - 提供“立即支付”与“暂存订单”两种模式，暂存订单将保留 15 分钟并在倒计时结束后自动释放座位。

4. **订单生命周期**  
   - 用户可在“我的订票单”中查看所有订单，支持取消、申请退款、查看改签记录。  
   - 当用户发起改签时，系统会加载原订单，保持原舱位与票数并计算新旧航班差价，若需补差则直接从钱包扣除。

5. **AI 客服与人工客服协同**  
   - 在用户端 Header 与客服页面集成“AI 智能客服”入口，基于大语言模型的对话代理能够理解航班查询、订单状态、退款政策等自然语言问题，提供实时答案。  
   - AI 会结合当前用户会话上下文（如最近查询航班、未完成订单）生成个性化建议，并可以直接调用 flightApi/search 接口返回的结构化数据以 Markdown 表格形式展示。  
   - 当 AI 识别到“高投诉/复杂问题/用户要求人工”等意图时，会通过客服调度接口开启“人工客服接管”流程：在 LiveCustomer 页面中展示人工客服状态，用户可与真人客服实时聊天；整个流程会记录在 support_tickets 表中，便于审计与回溯。

6. **工单与服务追踪**  
   - Contact 页面允许用户提交工单，系统自动填充用户昵称、联系方式，可选择优先级。  
   - “我的工单”页面根据是否存在管理员回复显示“未受理/已受理”，颜色区分清晰，支持按状态过滤与查看时间线。

### 3.2 管理员端能力

1. **登录与权限校验**  
   - 管理后台在 `useEffect` 中检查当前登录用户，若非 ADMIN 则跳转登录页。  
   - 所有后端接口均在 Controller 层进行角色校验，防止越权访问。

2. **仪表盘与统计分析**  
   - Dashboard 集成 ReactECharts，支持柱状/折线/饼图切换，展示订单趋势、舱位销售占比、用户增长等指标。  
   - 通过 `userApi.stats()` 获取活跃用户、VIP 用户、钱包余额总量等统计信息。

3. **航班管理**  
   - 航班列表按创建时间倒序排列，表格展示航线、时间区间、最低票价、余票、操作按钮。  
   - 新增/编辑抽屉中，机型选择可以触发 AIRCRAFT_PRESETS 自动填充座位与票价，并允许细粒度调整；城市选择组件复用用户端逻辑，保证体验一致。  
   - 校验规则包括：必填字段、到达时间必须晚于起飞时间、出发地与目的地不得相同、票价与座位数需为正数、余票不大于总座位数。  
   - 保存后调用 flightApi，成功则刷新列表并给出 Toast 提示；删除航班时会二次确认，并触发后端级联删除座位与订单。

4. **订单管理**  
   - 支持按状态、时间筛选；列表默认按创建时间倒序。  
   - 提供退款审核弹窗，管理员可选择“同意/驳回”，填写原因与手续费，系统将同步更新订单状态并记录操作人。  
   - “查看订单”按钮可打开 Drawer，展示该航班的所有订单、总票数、营收等统计数据。

5. **用户管理**  
   - Administrators 可检索用户，查看钱包余额、账号状态，支持禁用/启用操作。  
   - 对于钱包异常情况可通过 `WalletRechargeRequest` 进行补充值，所有变动都会写入日志。

6. **工单中心**  
   - 提供状态筛选、分页浏览、按优先级排序等功能；管理员可直接在工单详情页回复用户，并将状态更新为 IN_PROGRESS、RESOLVED 等。  
   - 工单回复同时会推送消息到用户“我的工单”页，并通知 AI 客服更新上下文，以便其在后续对话中引用最新处理结果。

7. **AI 客服训练与人工模拟**  
   - 管理后台内置“AI 客服训练”页面，可录入常见问题、标准答案、流程脚本；这些资料将同步到 AI 向量库，以提升回答准确度。  
   - 还提供“人工客服模拟 AI”模式：管理员可以在测试环境中扮演 AI，模拟回答逻辑，为模型上线前做 AB 测试。此模式会记录问答对，并可导出训练语料。

### 3.3 支撑服务

1. **定时任务与数据清洗**  
   - `purgeLegacyFlightsOnStartup()`：启动阶段自动删除 flight_number 为“999”或 arrival≤departure、票价为零的航班，确保接口返回的数据均有效。  
   - `restoreSeatsForBooking()`：在订单取消时针对 legacy 航班缺少 FlightSeat 的情况进行兜底处理，防止抛出异常。  
   - 座位汇总：通过 SeatRepository 与 BookingSeatRepository 的聚合查询，实时构建 cabin 级别的余票统计，用于前端展示与折扣策略。

2. **数据库迁移与演练脚本**  
   - `database/migration_add_flight_remarks.sql` 用于新增 remarks 字段，并兼容已有数据。  
   - `database/1.sql` 提供完整初始化数据，包括机型、航班、订单、座位、用户等，可直接在 Navicat 中运行。  
   - 文档中附带“批量新增航班并模拟多用户购票”的 SQL 模板，涵盖新增航班、批量初始化座位、让多名非管理员用户购买不同数量机票等步骤，确保测试环境数据多样化。

## 4. 数据模型与业务规则

1. **航班 (Flight)**：字段包括 flightNumber、aircraftType、origin、destination、departureTime、arrivalTime、remarks；Service 层保证 flight_seats 与 seats 间引用不为空，并在保存时调用 `prepareSeatEntities()` 设置 `seat.setFlight(this)`。  
2. **座位 (Seat)**：采用 `flight_id + seat_number` 唯一约束，记录 cabinClass、row、column、seatType、price_surcharge 等信息；配合存储过程按机型自动生成。  
3. **订单 (Booking)**：包含 status/ticketCount/totalAmount/cabinClass/paymentDueAt/refundFee 等字段；改签、退款等扩展字段记录历史信息，方便审计。  
4. **钱包 (User.wallet_balance)**：所有扣款/退款/改签补差都直接作用于钱包余额，前端通过 `saveAuth` 同步最新余额。  
5. **AI 客服会话 (SupportTicket + Conversation Logs)**：工单表扩展字段记录 AI 是否介入、人工接管次数、模型版本等，可用于后续模型评估。

业务规则概述：  
- 航班的数据有效性需满足“时间合法 + 票价正数 + 至少一个舱位启用”；  
- 订单创建后 15 分钟未支付则自动取消释放座位；  
- 改签差价计算包含“新航班金额 + 手续费 − 原航班退款”；  
- AI 客服回复时如触发调用航班接口，需要校验请求参数避免 SQL 注入；  
- 管理员新增航班时，出发地与目的地不允许相同，否则在 validateFlightForm 中直接报错。

## 5. AI 客服与人工客服协同机制

AI 客服系统采用“对话意图识别 + 工具调用 + 人工接管”的三阶段流程：

1. **意图识别**：前端通过 WebSocket 将用户输入发送给 AI 网关，模型对文本做意图分类（查询航班、订单状态、投诉、需要人工等），并提取关键信息（时间、城市、订单号）。  
2. **工具调用**：当意图为航班查询时，AI 会调用 `flightApi.search` 或 `flightApi.listAvailable` 获取实时数据，再将结构化信息转为自然语言答案；若为订单相关问题，则调用 `bookingApi.list` 或 `ticketApi.list` 提供状态反馈。  
3. **人工接管**：若模型置信度不足或用户明确要求人工协助，系统会将当前会话上下文推送给 LiveCustomer 模块，人工客服即可阅读 AI 历史回答并继续对话；人工回复同时写回对话日志，使 AI 可以在未来引用人工经验。  
4. **训练闭环**：管理员可导出问答日志，对 AI 答案进行标注、纠错，并通过训练页面将高质量语料送入模型，形成持续改进闭环。该策略在实验环境中证实可将客服首次响应时间降低 60% 以上。

## 6. 性能与可靠性优化

1. **座位汇总查询**：改用聚合 SQL（COUNT/SUM）代替逐条加载 seats，显著降低航班列表接口的响应时间。  
2. **Discount Cache**：对折扣航班排行使用缓存保留策略，避免列表频繁抖动，提升用户感知稳定性。  
3. **CityPicker 节流**：通过 200ms 防抖与 Portal 渲染，兼顾输入响应速度与弹出效果。  
4. **错误兜底机制**：当 flightApi.listAvailable 接口异常时，前端自动回退至 search 接口并展示告警，而不会阻断用户操作。  
5. **AI 客服监控**：对接管率、置信度、平均响应时间等指标进行采集，管理员可在仪表盘查看 AI 服务质量，必要时调整路由策略。

## 7. 未来扩展方向

1. **动态收益管理**：结合实时座位、预订节奏与历史价格引入机器学习模型，实现更细粒度的票价预测。  
2. **多渠道接入**：目前 AI 客服主要服务 Web 端，后续可扩展到小程序、App、电话机器人，实现全渠道一致体验。  
3. **数据湖与实时分析**：可将航班、订单、客服日志同步至数据湖或 Kafka，构建实时监控与 BI 报表，更好地支持经营决策。  
4. **隐私与合规**：对涉及身份证、联系方式等敏感信息引入脱敏策略，并记录所有数据访问行为，满足航空业合规要求。

## 8. 结论

本项目通过统一的技术底座与严格的业务规则，构建了一套涵盖航班管理、票务销售、钱包支付、工单处置、AI 客服以及人工客服协同的全栈航空服务平台。系统在实践中证明：  
1. 通过双座位模型与自动清理机制，旧数据向新系统迁移的复杂度显著降低；  
2. 票价折扣、紧急加价与改签差价的组合策略，在保障旅客体验的同时提升了收益管控能力；  
3. AI 客服与人工客服的协同，使平台实现了“机器人 7×24 小时在线 + 人工精准接入”的服务形态，极大降低人力成本；  
4. 前后端一致的输入校验与状态同步机制，确保数据可靠与交互流畅；  
5. 丰富的管理员功能（航班管理、订单管理、工单中心、用户管理、AI 训练）使平台适配多角色协作需求。

综合来看，该平台既满足航空企业在运营层面的高可用、高安全与高可扩展需求，也为研究者提供了一个观察 AI 与传统业务深度融合的实验场景。未来可继续围绕智能收益管理、全渠道客服、实时数据分析等方向深化，推动航空数字化持续演进。
