# 一、系统架构相关问题

## 文档说明

本文档详细回答 ZJC 航空公司机票预订系统的架构相关问题，包括架构设计、设计模式、前后端分离、API 设计原则以及系统可扩展性和可维护性。

**系统版本：** v1.0  
**文档日期：** 2025-11-20  
**技术栈：** Spring Boot 3.3.0 + Next.js 14 + MySQL 8.0+

---

## 1. 请简述该机票预订系统的架构设计，包括使用了哪些层级？

### 1.1 整体架构概述

ZJC 航空公司机票预订系统采用**前后端分离的三层架构模式**，结合经典的 MVC（Model-View-Controller）设计思想，实现了表现层、业务逻辑层和数据持久层的清晰分离。

### 1.2 架构层级详解

#### 1.2.1 表现层（Presentation Layer）

**前端架构：**
- **技术栈：** Next.js 14.2.5 (React 18.3.1) + TypeScript 5
- **UI 组件库：** Chakra UI 2.8.2
- **状态管理：** React Hooks + React Redux
- **HTTP 客户端：** Axios 1.7.7
- **路由：** Next.js App Router

**职责：**
- 用户界面的渲染和交互
- 用户输入的收集和验证
- 通过 RESTful API 与后端进行数据交互
- 实现响应式设计，支持多设备访问
- 前端路由守卫和权限控制

**文件结构：**
```
zjc-airline-booking-frontend/src/
├── app/                  # Next.js App Router 页面
│   ├── admin/           # 管理员面板
│   ├── bookings/        # 订票单管理
│   ├── flights/         # 航班查询
│   ├── login/           # 登录页面
│   └── register/        # 注册页面
├── components/           # React 组件
│   ├── layout/          # 布局组件（Header、Footer）
│   └── common/          # 通用组件
└── lib/                 # 工具库（API 客户端、认证等）
```

#### 1.2.2 业务逻辑层（Business Logic Layer）

**后端架构：**
- **框架：** Spring Boot 3.3.0
- **编程语言：** Java 17
- **API 规范：** RESTful API
- **构建工具：** Maven

**层级结构：**

**Controller 层（控制器层）**
- **位置：** `com.example.demo.controller`
- **职责：**
  - 接收和处理 HTTP 请求
  - 参数验证和转换
  - 调用 Service 层处理业务逻辑
  - 返回 JSON 格式的响应
- **示例类：**
  - `FlightController`：处理航班相关请求
  - `BookingController`：处理订单相关请求
  - `UserController`：处理用户相关请求
  - `SeatController`：处理座位相关请求

**Service 层（业务逻辑层）**
- **位置：** `com.example.demo.service`
- **职责：**
  - 核心业务逻辑处理
  - 数据验证和业务规则执行
  - 事务管理（`@Transactional`）
  - 调用 Repository 层进行数据访问
  - 跨实体的业务协调
- **示例类：**
  - `FlightService`：航班管理业务逻辑
  - `BookingService`：订单管理业务逻辑（预订、退票、改签）
  - `UserService`：用户管理业务逻辑
  - `SeatService`：座位分配业务逻辑

**DTO 层（数据传输对象层）**
- **位置：** `com.example.demo.dto`
- **职责：**
  - 定义 API 请求和响应的数据结构
  - 实现前后端数据传输的契约
  - 数据验证和类型安全
- **示例类：**
  - `FlightSearchRequest`：航班搜索请求
  - `BookingRequest`：预订请求
  - `ApiResponse<T>`：统一响应格式

#### 1.2.3 数据持久层（Data Persistence Layer）

**Repository 层（数据访问层）**
- **位置：** `com.example.demo.repository`
- **技术：** Spring Data JPA + Hibernate
- **职责：**
  - 封装数据库访问操作
  - 提供 CRUD 操作接口
  - 自定义查询方法
  - 数据查询优化
- **示例接口：**
  - `FlightRepository`：航班数据访问
  - `BookingRepository`：订单数据访问
  - `UserRepository`：用户数据访问

**Entity 层（实体层）**
- **位置：** `com.example.demo.entity`
- **技术：** JPA Entity + Hibernate ORM
- **职责：**
  - 定义数据库表映射
  - 实体关系映射（一对一、一对多、多对多）
  - 数据验证注解
  - 生命周期回调
- **示例类：**
  - `Flight`：航班实体
  - `Booking`：订单实体
  - `User`：用户实体
  - `Seat`：座位实体

**数据库层**
- **数据库：** MySQL 8.0+
- **存储引擎：** InnoDB
- **字符集：** utf8mb4
- **职责：**
  - 持久化存储业务数据
  - 数据完整性约束（外键、唯一性、非空）
  - 索引优化查询性能
  - 事务支持

### 1.3 数据流设计

**请求流程：**
```
浏览器 → Next.js 前端 → Axios HTTP 请求 → Spring Boot Controller
→ Service 业务逻辑处理 → Repository 数据访问 → MySQL 数据库
```

**响应流程：**
```
MySQL 数据库 → Repository → Service → Controller → JSON 响应
→ 前端组件渲染 → 用户界面更新
```

**认证流程：**
```
用户登录 → 后端验证（UserService） → 返回用户信息 → 
前端存储到 LocalStorage → 后续请求携带用户信息
```

## 2. 系统采用了哪些设计模式？为什么选择这些设计模式？

### 2.1 采用的设计模式

#### 2.1.1 MVC 模式（Model-View-Controller）

**应用位置：**
- **Model（模型）：** Entity 层 + Repository 层
- **View（视图）：** Next.js 前端页面组件
- **Controller（控制器）：** Spring Boot Controller 层

**优势：**
- 职责分离清晰，便于维护和测试
- 前后端解耦，可独立开发和部署
- 代码复用性高

#### 2.1.2 分层架构模式（Layered Architecture）

**应用位置：**
- Controller → Service → Repository → Database

**优势：**
- 每一层专注于自己的职责
- 便于单元测试和集成测试
- 降低层间耦合，提高系统可扩展性

#### 2.1.3 Repository 模式

**应用位置：**
- 所有 `*Repository` 接口（如 `FlightRepository`、`BookingRepository`）

**实现方式：**
- 继承 `JpaRepository<T, ID>` 接口
- 自动提供 CRUD 操作
- 支持自定义查询方法（通过方法名约定或 `@Query` 注解）

**优势：**
- 隐藏数据访问实现细节
- 统一数据访问接口
- 便于切换不同的数据源（如从 MySQL 切换到 PostgreSQL）

**代码示例：**
```java
public interface FlightRepository extends JpaRepository<Flight, Long> {
    // 自动提供：save, findById, findAll, delete 等方法
    
    // 自定义查询方法（通过方法名约定）
    Optional<Flight> findByFlightNumber(String flightNumber);
    List<Flight> findByOriginContainingIgnoreCase(String origin);
    
    // 使用 @Query 注解自定义 SQL
    @Query("SELECT f FROM Flight f WHERE f.departureTime > :now")
    List<Flight> findAvailableFlights(@Param("now") LocalDateTime now);
}
```

#### 2.1.4 DTO 模式（Data Transfer Object）

**应用位置：**
- 所有 `*Request` 和 `*Response` 类（如 `BookingRequest`、`FlightSearchRequest`）

**优势：**
- 前后端数据契约明确
- 避免直接暴露 Entity 内部结构
- 支持数据验证和转换
- 减少网络传输数据量

#### 2.1.5 依赖注入模式（Dependency Injection）

**应用位置：**
- 所有 `@Autowired` 注解的使用（如 Service 注入 Repository，Controller 注入 Service）

**实现方式：**
- Spring 的 `@Autowired` 注解
- 构造函数注入（推荐）
- 字段注入

**优势：**
- 降低类之间的耦合
- 便于单元测试（可轻松 mock 依赖）
- 提高代码可维护性

#### 2.1.6 单例模式（Singleton）

**应用位置：**
- Spring Bean 默认就是单例模式

**优势：**
- 节省内存资源
- 提高性能（避免重复创建对象）

#### 2.1.7 工厂模式（Factory Pattern）

**应用位置：**
- Spring 容器本身就是 Bean 工厂
- `PasswordEncoder` Bean 的创建（`BCryptPasswordEncoder`）

**优势：**
- 统一对象创建逻辑
- 便于管理和配置

### 2.2 选择这些设计模式的原因

**1. 提高代码质量**
- 设计模式使代码结构清晰，易于理解和维护
- 遵循行业最佳实践，降低开发风险

**2. 提升开发效率**
- Spring Framework 内置支持这些模式，开发便捷
- 减少重复代码，提高代码复用性

**3. 增强系统可扩展性**
- 分层架构便于添加新功能
- Repository 模式便于切换数据源
- DTO 模式便于 API 版本升级

**4. 便于测试和维护**
- 依赖注入便于编写单元测试
- 分层架构便于定位问题
- 设计模式使代码更易于重构

## 3. 前后端是如何分离的？使用了哪些技术栈？

### 3.1 前后端分离架构

#### 3.1.1 分离方式

**完全独立部署：**
- **前端：** 独立的 Next.js 应用，运行在 `http://localhost:3000`
- **后端：** 独立的 Spring Boot 应用，运行在 `http://localhost:8080`
- **通信方式：** HTTP/HTTPS + RESTful API

**数据交换格式：**
- **请求格式：** JSON（Content-Type: application/json）
- **响应格式：** JSON（统一封装在 `ApiResponse<T>` 中）

#### 3.1.2 前后端分离的优势

1. **独立开发和部署**
   - 前端和后端可以并行开发
   - 可以独立进行版本控制和发布
   - 便于团队协作（前端团队和后端团队可以独立工作）

2. **技术栈解耦**
   - 前端可以使用 React、Vue 等任意框架
   - 后端可以使用 Spring Boot、Node.js 等任意技术
   - 技术选型更加灵活

3. **性能优化**
   - 前端可以部署到 CDN，提高访问速度
   - 后端可以独立进行水平扩展
   - 前后端可以分别进行性能优化

4. **安全性提升**
   - API 接口可以统一进行安全控制
   - 前后端可以分别进行安全加固

## 4. 系统的 API 设计遵循了什么原则？

### 4.1 RESTful API 设计原则

#### 4.1.1 资源导向

**URL 设计：**
- 使用名词表示资源，不使用动词
- 使用复数形式表示资源集合
- 示例：
  - ✅ `GET /api/flights`：获取航班列表
  - ✅ `GET /api/flights/{id}`：获取单个航班
  - ✅ `POST /api/bookings`：创建订单
  - ❌ `GET /api/getFlights`（不推荐，包含动词）

#### 4.1.2 HTTP 方法语义化

**GET：** 查询资源（幂等、安全）
- `GET /api/flights`：查询航班列表
- `GET /api/bookings/{id}`：查询订单详情

**POST：** 创建资源（非幂等）
- `POST /api/bookings`：创建订单
- `POST /api/users`：注册用户

**PUT/PATCH：** 更新资源（幂等）
- `PUT /api/flights/{id}`：更新航班信息
- `PATCH /api/users/{id}`：部分更新用户信息

**DELETE：** 删除资源（幂等）
- `DELETE /api/flights/{id}`：删除航班

#### 4.1.3 状态码规范

**2xx 成功：**
- `200 OK`：请求成功
- `201 Created`：资源创建成功

**4xx 客户端错误：**
- `400 Bad Request`：请求参数错误
- `401 Unauthorized`：未认证
- `403 Forbidden`：无权限
- `404 Not Found`：资源不存在

**5xx 服务器错误：**
- `500 Internal Server Error`：服务器内部错误

#### 4.1.4 统一响应格式

**响应结构：**
```java
// 后端 ApiResponse 类
public class ApiResponse<T> {
    private boolean success;
    private T data;
    private String message;
    private String error;
    
    // 静态工厂方法
    public static <T> ApiResponse<T> success(T data) {
        ApiResponse<T> response = new ApiResponse<>();
        response.setSuccess(true);
        response.setData(data);
        return response;
    }
    
    public static <T> ApiResponse<T> error(String message) {
        ApiResponse<T> response = new ApiResponse<>();
        response.setSuccess(false);
        response.setError(message);
        return response;
    }
}
```

**使用示例：**
```java
@GetMapping("/{id}")
public ResponseEntity<ApiResponse<Flight>> getFlightById(@PathVariable Long id) {
    try {
        Flight flight = flightService.getFlightById(id);
        return ResponseEntity.ok(ApiResponse.success(flight));
    } catch (Exception e) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.error(e.getMessage()));
    }
}
```

### 4.2 API 设计最佳实践

#### 4.2.1 版本控制

**URL 版本控制（当前实现）：**
- 所有 API 使用 `/api` 前缀
- 未来如需版本控制，可使用 `/api/v1/`、`/api/v2/` 等

#### 4.2.2 分页和排序

**分页参数：**
- `page`：页码（从 0 开始）
- `size`：每页大小
- 示例：`GET /api/flights?page=0&size=10`

**排序参数：**
- `sort`：排序字段和方向
- 示例：`GET /api/flights?sort=departureTime,desc`

#### 4.2.3 过滤和搜索

**查询参数：**
- 使用查询参数进行过滤
- 示例：`GET /api/flights?origin=北京&destination=上海`

#### 4.2.4 错误处理

**统一异常处理：**
- 使用 `@ControllerAdvice` 全局异常处理
- 返回统一的错误响应格式
- 记录错误日志

#### 4.2.5 API 文档

**文档工具：**
- 提供 OpenAPI 规范文档（`api/openapi.yaml`）
- 提供 Postman Collection（`api/ZJC_Airline_Booking_API.postman_collection.json`）
- 提供 Markdown 文档（`api/API_DOCUMENTATION.md`）

## 5. 如何保证系统的可扩展性和可维护性？

### 5.1 可扩展性设计

#### 5.1.1 模块化设计

**分层架构：**
- Controller、Service、Repository 各层职责清晰
- 各层可以独立扩展和优化

**组件化：**
- 前端使用 React 组件化开发
- 后端 Service 按业务模块划分（FlightService、BookingService 等）

#### 5.1.2 数据库扩展性

**索引优化：**
- 为常用查询字段创建索引
- 支持复合索引优化多条件查询

**分表策略：**
- 未来可按时间或业务范围进行分表
- 使用数据库分区提高查询性能

**读写分离：**
- 未来可实现主从复制，读写分离
- 提高数据库并发处理能力

#### 5.1.3 水平扩展

**无状态设计：**
- 后端服务无状态，便于水平扩展
- 会话信息存储在客户端（LocalStorage）

**负载均衡：**
- 未来可使用 Nginx 或 Spring Cloud Gateway 进行负载均衡
- 支持多实例部署

#### 5.1.4 缓存机制

**当前实现：**
- Hibernate 二级缓存（可选）
- 前端 LocalStorage 缓存用户信息

**未来扩展：**
- 引入 Redis 缓存热点数据（航班信息、座位信息）
- 减少数据库查询压力

#### 5.1.5 微服务架构（未来）

**服务拆分：**
- 可按业务模块拆分为微服务：
  - 用户服务（User Service）
  - 航班服务（Flight Service）
  - 订单服务（Booking Service）
  - 座位服务（Seat Service）

**服务通信：**
- 使用 RESTful API 或消息队列（如 RabbitMQ、Kafka）

### 5.2 可维护性设计

#### 5.2.1 代码规范

**命名规范：**
- 类名使用大驼峰（PascalCase）：`FlightService`
- 方法名使用小驼峰（camelCase）：`getFlightById`
- 常量使用大写下划线（UPPER_SNAKE_CASE）：`URGENT_WINDOW_HOURS`

**注释规范：**
- 类和方法添加 JavaDoc 注释
- 复杂业务逻辑添加行内注释

#### 5.2.2 单一职责原则

**Service 层设计：**
- 每个 Service 类只负责一个业务领域
- 例如：`FlightService` 只处理航班相关业务，`BookingService` 只处理订单相关业务

#### 5.2.3 DRY 原则（Don't Repeat Yourself）

**工具类抽取：**
- 公共逻辑抽取为工具类（如 `PasswordGenerator`）
- 前端公共逻辑抽取为自定义 Hooks 或工具函数

#### 5.2.4 配置外部化

**配置文件：**
- 数据库连接、端口等配置放在 `application.properties`
- 敏感信息使用环境变量或配置中心

**代码示例：**
```properties
# application.properties
spring.datasource.url=jdbc:mysql://localhost:3306/zjc_airline_booking_db
spring.datasource.username=root
spring.datasource.password=${DB_PASSWORD}
server.port=8080
```

#### 5.2.5 日志记录

**日志框架：**
- 使用 SLF4J + Logback
- 记录关键操作和异常信息

**日志级别：**
- `DEBUG`：调试信息
- `INFO`：一般信息（如业务操作）
- `WARN`：警告信息（如数据不一致）
- `ERROR`：错误信息（如异常）

**代码示例：**
```java
private static final Logger log = LoggerFactory.getLogger(FlightService.class);

public void createFlight(Flight flight) {
    log.info("创建航班: {}", flight.getFlightNumber());
    try {
        flightRepository.save(flight);
        log.info("航班创建成功: {}", flight.getFlightNumber());
    } catch (Exception e) {
        log.error("航班创建失败: {}", e.getMessage(), e);
        throw new RuntimeException("航班创建失败", e);
    }
}
```
