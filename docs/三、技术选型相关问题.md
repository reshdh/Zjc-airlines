# 三、技术选型相关问题

## 文档说明

本文档详细回答 ZJC 航空公司机票预订系统的技术选型相关问题，包括为什么选择 Spring Boot、JPA、MySQL、Next.js、RESTful API 等技术栈，以及各自的优势和特点。

**系统版本：** v1.0  
**文档日期：** 2025-11-20  
**技术栈：** Spring Boot 3.3.0 + Next.js 14 + MySQL 8.0+

## 1. 为什么选择 Spring Boot 作为后端框架？

### 1.1 Spring Boot 的优势

#### 1.1.1 约定优于配置（Convention over Configuration）

**特点：**
- Spring Boot 提供了大量默认配置，减少配置工作量
- 使用约定来减少决策，提高开发效率
- 开箱即用，快速启动项目

**示例：**
- 默认嵌入 Tomcat 服务器，无需单独部署
- 默认配置数据库连接池
- 自动配置 JPA、Security 等常用组件

#### 1.1.2 自动配置（Auto-Configuration）

**特性：**
- 根据类路径中的依赖自动配置 Spring 应用
- 智能检测并配置所需的 Bean
- 提供大量 Starter 依赖简化项目配置

**Starter 示例：**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
<!-- 自动包含：Spring MVC、Tomcat、Jackson 等 -->
```

#### 1.1.3 生产级特性

**内置功能：**
- **监控和健康检查：** Spring Boot Actuator
- **应用指标：** 提供丰富的运行时指标
- **外部化配置：** 支持多环境配置（dev、test、prod）
- **日志管理：** 默认集成 Logback

#### 1.1.4 简化的依赖管理

**Maven/Gradle 集成：**
- Parent POM 管理版本依赖，避免版本冲突
- 依赖版本自动协调
- 减少依赖冲突问题

#### 1.1.5 快速开发

**开发效率：**
- 快速创建项目（Spring Initializr）
- 热部署支持（Spring Boot DevTools）
- 丰富的开发工具和插件支持

### 1.2 与其他框架的对比

#### vs 传统 Spring Framework

**Spring Boot 优势：**
- ✅ 无需繁琐的 XML 配置
- ✅ 内置服务器，部署简单
- ✅ 自动配置，开发更快
- ✅ 生产级特性开箱即用

**传统 Spring：**
- ❌ 需要大量 XML 配置
- ❌ 需要单独部署 Tomcat
- ❌ 需要手动配置各个组件
- ❌ 需要额外配置监控工具

#### vs Node.js（Express/Koa）

**Spring Boot 优势：**
- ✅ Java 生态成熟，库丰富
- ✅ 强类型语言，减少运行时错误
- ✅ 企业级应用广泛使用
- ✅ JVM 性能优化成熟

**Node.js 优势：**
- ✅ 异步 I/O 性能优秀
- ✅ 前后端统一语言
- ✅ 轻量级，启动快

**选择原因：**
- 团队熟悉 Java
- 需要处理复杂的业务逻辑
- 企业级应用要求稳定性

### 1.3 项目中的应用

**版本选择：**
- Spring Boot 3.3.0（最新稳定版）
- Java 17（LTS 版本）

**使用的 Starter：**
```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-data-jpa</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-security</artifactId>
    </dependency>
</dependencies>
```

## 2. 为什么选择 JPA 作为持久层框架？相比 MyBatis 有什么优势？

### 2.1 JPA（Java Persistence API）的优势

#### 2.1.1 ORM（对象关系映射）优势

**特点：**
- 将数据库表映射为 Java 对象（Entity）
- 自动处理 SQL 生成
- 减少手写 SQL 代码

**示例：**
```java
@Entity
@Table(name = "flights")
public class Flight {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(nullable = false)
    private String flightNumber;
    
    // JPA 自动生成 SQL：SELECT * FROM flights WHERE id = ?
}
```

#### 2.1.2 Spring Data JPA 的便利性

**Repository 接口：**
```java
public interface FlightRepository extends JpaRepository<Flight, Long> {
    // 方法名自动生成查询（无需手写 SQL）
    Optional<Flight> findByFlightNumber(String flightNumber);
    List<Flight> findByOriginContainingIgnoreCase(String origin);
    
    // 使用 @Query 注解自定义查询
    @Query("SELECT f FROM Flight f WHERE f.departureTime > :now")
    List<Flight> findAvailableFlights(@Param("now") LocalDateTime now);
}
```

**优势：**
- 方法名约定自动生成查询
- 减少大量样板代码
- 类型安全，编译时检查

#### 2.1.3 关系映射自动化

**实体关系：**
```java
@Entity
public class Booking {
    @ManyToOne
    @JoinColumn(name = "flight_id")
    private Flight flight;  // JPA 自动处理外键关系
    
    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;
    
    @OneToMany(mappedBy = "booking", cascade = CascadeType.ALL)
    private List<BookingSeat> bookingSeats;
}
```

**优势：**
- 自动管理外键关系
- 支持级联操作（CASCADE）
- 延迟加载（Lazy Loading）

#### 2.1.4 事务管理集成

**声明式事务：**
```java
@Service
@Transactional  // 自动管理事务
public class BookingService {
    public Booking createBooking(BookingRequest request) {
        // 方法内所有数据库操作在同一事务中
        // 出现异常自动回滚
    }
}
```

### 2.2 JPA vs MyBatis

#### 2.2.1 开发效率对比

**JPA 优势：**
- ✅ 快速开发，减少 SQL 编写
- ✅ 方法名约定自动生成查询
- ✅ 关系映射自动化
- ✅ 代码量更少

**MyBatis 特点：**
- ✅ 需要手写 SQL（更灵活）
- ✅ 需要手写 XML 映射文件
- ✅ 对复杂 SQL 支持更好
- ❌ 代码量更多

#### 2.2.2 性能对比

**JPA：**
- Hibernate 提供了多种优化策略
- 一级缓存、二级缓存支持
- 延迟加载减少不必要的查询
- 批量操作优化

**MyBatis：**
- 直接 SQL，性能可控
- 手动控制缓存策略
- 对复杂查询性能更优

#### 2.2.3 适用场景

**JPA 适合：**
- ✅ CRUD 操作较多
- ✅ 关系复杂的领域模型
- ✅ 快速开发原型
- ✅ 团队熟悉 ORM

**MyBatis 适合：**
- ✅ 复杂 SQL 查询较多
- ✅ 需要精细控制 SQL
- ✅ 性能要求极高
- ✅ 遗留系统迁移

### 2.3 项目中选择 JPA 的原因

**原因：**
1. **快速开发：** 项目需要快速迭代，JPA 减少代码量
2. **业务模型复杂：** 航班、订单、用户、座位等实体关系复杂，JPA 关系映射更便利
3. **团队熟悉：** 团队更熟悉 ORM 框架
4. **Spring 生态：** 与 Spring Boot 集成更好

**项目中的应用：**
```java
// 实体定义
@Entity
public class Flight { /* ... */ }

// Repository 接口
public interface FlightRepository extends JpaRepository<Flight, Long> {
    // 方法名自动生成查询
}

// Service 层使用
@Autowired
private FlightRepository flightRepository;

public Flight getFlightById(Long id) {
    return flightRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("航班不存在"));
}
```

## 3. 为什么选择 MySQL 作为数据库？

### 3.1 MySQL 的优势

#### 3.1.1 成熟稳定

**特点：**
- 历史悠久，广泛应用于生产环境
- 大量实际应用验证，稳定性高
- 社区活跃，问题解决及时

#### 3.1.2 性能优秀

**优势：**
- 查询性能优秀
- 支持索引优化
- 支持分区表提高性能
- InnoDB 存储引擎支持行级锁

#### 3.1.3 成本效益

**优势：**
- 开源免费（社区版）
- 商业版价格合理
- 运维成本低
- 学习资源丰富

#### 3.1.4 功能丰富

**特性：**
- 支持 ACID 事务
- 支持外键约束
- 支持视图、存储过程、触发器
- 支持主从复制、读写分离

#### 3.1.5 生态支持

**工具支持：**
- MySQL Workbench（官方图形工具）
- Navicat（第三方工具）
- phpMyAdmin（Web 管理工具）
- 丰富的监控和备份工具

### 3.2 与其他数据库的对比

#### vs PostgreSQL

**MySQL 优势：**
- ✅ 学习曲线更平缓
- ✅ 运维经验更丰富
- ✅ 社区资源更多
- ✅ 工具支持更完善

**PostgreSQL 优势：**
- ✅ 功能更强大（JSON、全文搜索等）
- ✅ 标准 SQL 支持更好
- ✅ 复杂查询性能更优

#### vs MongoDB（NoSQL）

**MySQL（关系型）优势：**
- ✅ ACID 事务保证
- ✅ 数据一致性更强
- ✅ 复杂的关联查询
- ✅ SQL 标准支持

**MongoDB（NoSQL）优势：**
- ✅ 灵活的文档模型
- ✅ 水平扩展更容易
- ✅ 读写性能高（特定场景）

**选择 MySQL 的原因：**
- 数据关系明确（航班、订单、用户关联）
- 需要事务保证（订单创建、退款等）
- 团队更熟悉关系型数据库

### 3.3 项目中的配置

**版本：**
- MySQL 8.0+（支持新特性：JSON、窗口函数等）

**配置：**
```properties

spring.datasource.username=root
spring.datasource.password=root
```

**字符集：**
- utf8mb4（支持完整的 UTF-8，包括 Emoji）

## 5. 为什么使用 RESTful API？有什么优缺点？

### 5.1 RESTful API 的优势

#### 5.1.1 标准化设计

**原则：**
- 使用标准 HTTP 方法（GET、POST、PUT、DELETE）
- 使用标准 HTTP 状态码（200、201、400、404、500）
- URL 资源导向（使用名词，不使用动词）
- 无状态通信

**示例：**
```
GET    /api/flights          # 获取航班列表
GET    /api/flights/1        # 获取单个航班
POST   /api/bookings         # 创建订单
PUT    /api/bookings/1       # 更新订单
DELETE /api/flights/1        # 删除航班
```

#### 5.1.2 易于理解和维护

**优势：**
- URL 直观，易于理解
- HTTP 方法语义明确
- 遵循行业标准，易于对接
- 文档清晰，易于维护

#### 5.1.3 前后端解耦

**优势：**
- 前后端可以独立开发
- 前后端可以独立部署
- 支持多种客户端（Web、移动端、第三方系统）
- 易于版本控制

#### 5.1.4 缓存支持

**HTTP 缓存：**
- 使用 HTTP 缓存头（Cache-Control、ETag）
- 浏览器自动缓存 GET 请求
- CDN 缓存支持
- 减少服务器压力

### 5.2 RESTful API 的缺点

#### 5.2.1 过度获取/不足获取（Over-fetching/Under-fetching）

**问题：**
- 客户端可能需要多次请求才能获取完整数据
- 可能返回客户端不需要的字段
- GraphQL 可以解决这个问题

**示例：**
```javascript
// RESTful：需要多次请求
GET /api/bookings/1      // 获取订单基本信息
GET /api/bookings/1/seats // 获取订单座位信息
GET /api/flights/1       // 获取航班信息

// GraphQL：一次请求获取所有数据
query {
  booking(id: 1) {
    id
    seats { ... }
    flight { ... }
  }
}
```

#### 5.2.2 版本控制

**问题：**
- API 版本管理需要考虑兼容性
- 旧版本 API 需要维护

**解决方案：**
- URL 版本控制：`/api/v1/flights`
- Header 版本控制：`Accept: application/vnd.api+json;version=1`
- 参数版本控制：`/api/flights?version=1`

#### 5.2.3 复杂查询

**问题：**
- 复杂查询需要多个参数
- URL 可能变得很长

**示例：**
```
GET /api/flights?origin=北京&destination=上海&date=2025-11-20&sort=price,asc&page=0&size=10
```

#### 5.2.4 实时通信

**问题：**
- RESTful API 是基于请求-响应模式
- 不支持服务器主动推送（需要 WebSocket）

**解决方案：**
- WebSocket 用于实时通信
- Server-Sent Events（SSE）用于服务器推送

### 5.3 项目中的 RESTful API 设计

**设计原则：**
- 资源导向：使用名词表示资源
- HTTP 方法语义化：GET 查询、POST 创建、PUT 更新、DELETE 删除
- 统一响应格式：`ApiResponse<T>`
- 状态码规范：200 成功、400 客户端错误、500 服务器错误

**API 示例：**
```java
@RestController
@RequestMapping("/api/flights")
public class FlightController {
    
    @GetMapping
    public ResponseEntity<ApiResponse<List<Flight>>> getAllFlights() {
        // 获取航班列表
    }
    
    @GetMapping("/{id}")
    public ResponseEntity<ApiResponse<Flight>> getFlightById(@PathVariable Long id) {
        // 获取单个航班
    }
    
    @PostMapping
    public ResponseEntity<ApiResponse<Flight>> createFlight(@RequestBody Flight flight) {
        // 创建航班
    }
}
```

**统一响应格式：**
```java
public class ApiResponse<T> {
    private boolean success;
    private T data;
    private String message;
    private String error;
}
```

### 5.4 替代方案

#### GraphQL

**优势：**
- 客户端可以精确指定需要的字段
- 一次请求获取多个资源
- 强类型系统

**劣势：**
- 学习曲线较陡
- 缓存机制不如 REST
- 复杂度较高

**适用场景：**
- 客户端需求多样化
- 移动端需要减少请求次数
- 复杂的数据关系

#### gRPC

**优势：**
- 高性能（基于 HTTP/2）
- 强类型（Protocol Buffers）
- 支持流式传输

**劣势：**
- 主要适用于服务间通信
- 浏览器支持有限

**适用场景：**
- 微服务间通信
- 高性能要求
- 内部服务调用
