# ZJC 航空公司机票预订系统 - 项目介绍文档

## 一、项目概述

### 1.1 项目简介

ZJC 航空公司机票预订系统是一个现代化的、功能完整的航空票务管理平台，旨在为用户提供便捷的航班查询、预订和管理服务，同时为管理员提供全面的数据管理和运营支持功能。该系统采用前后端分离的架构设计，实现了用户端和管理端的高效协同。

本系统支持完整的机票预订业务闭环，包括用户注册登录、航班信息查询、机票预订、订单管理、用户信息管理、工单反馈以及管理员对航班、订单、用户和工单的全面管理。系统设计注重用户体验、数据安全性和系统可扩展性，能够满足中小型航空公司的业务需求。

## 二、系统架构

### 2.1 整体架构

系统采用典型的三层架构模式，结合前后端分离的设计理念：

**表现层 (Presentation Layer)**
- 前端 Next.js 应用负责用户界面的渲染和交互
- 通过 RESTful API 与后端进行数据交互
- 实现了响应式设计，支持多设备访问

**业务逻辑层 (Business Logic Layer)**
- Spring Boot 后端服务处理业务逻辑
- Service 层负责核心业务处理
- Controller 层提供 RESTful API 接口

**数据持久层 (Data Persistence Layer)**
- JPA/Hibernate 实现 ORM 映射
- MySQL 数据库存储业务数据
- Repository 层封装数据访问操作

### 2.2 数据流设计

1. **用户请求流程**: 浏览器 → Next.js 前端 → Axios HTTP 请求 → Spring Boot 后端 → JPA Repository → MySQL 数据库
2. **响应返回流程**: MySQL → Repository → Service → Controller → JSON 响应 → 前端组件渲染
3. **认证流程**: 用户登录 → 后端验证 → 返回 Token → 前端存储 → 后续请求携带 Token

### 2.3 安全设计

- **密码加密**: 使用 BCryptPasswordEncoder 对用户密码进行哈希加密存储
- **CORS 配置**: 支持跨域请求，可在配置中限制允许的源
- **角色权限**: 实现用户角色（USER/ADMIN）区分，管理员功能需要相应权限
- **前端路由守卫**: 通过认证检查保护需要登录的页面

## 四、数据库设计

### 4.1 数据库概述

数据库采用 MySQL 8.0+，使用 InnoDB 存储引擎，字符集为 utf8mb4，支持完整的 Unicode 字符集。数据库设计遵循第三范式，确保数据的一致性和完整性。

### 4.2 数据表结构

#### users 表（用户表）
存储用户基本信息和认证信息：
- `id`: 主键，自增长
- `name`: 真实姓名
- `id_number`: 身份证号（唯一索引）
- `phone`: 手机号（索引）
- `address`: 通讯地址
- `username`: 用户名（唯一索引）
- `password`: 加密后的密码（BCrypt）
- `role`: 用户角色（USER/ADMIN）
- `status`: 账号状态（1-启用，0-禁用）
- `created_at`: 创建时间（自动维护）
- `updated_at`: 更新时间（自动维护）

#### flights 表（航班表）
存储航班信息和可用性：
- `id`: 主键，自增长
- `flight_number`: 航班号（唯一索引）
- `aircraft_type`: 机型
- `origin`: 出发地（索引）
- `destination`: 目的地（索引）
- `departure_time`: 出发时间（索引）
- `arrival_time`: 到达时间
- `price`: 票价（DECIMAL(10,2)）
- `remaining_seats`: 剩余座位数
- `remarks`: 备注信息
- `created_at`: 创建时间
- `updated_at`: 更新时间
- 复合索引：`(origin, destination)` 用于路线查询优化

#### bookings 表（预订表）
存储订单信息和状态：
- `id`: 主键，自增长
- `flight_id`: 航班ID（外键，关联 flights 表）
- `user_id`: 用户ID（外键，关联 users 表）
- `ticket_count`: 票数
- `total_amount`: 总金额（DECIMAL(10,2)）
- `status`: 订单状态（CREATED/PAID/CANCELED）
- `created_at`: 创建时间（索引，用于排序）
- `updated_at`: 更新时间
- 索引：`user_id`、`flight_id`、`status`、`created_at`

#### support_tickets 表（工单表）
存储用户反馈和管理员回复：
- `id`: 主键，自增长
- `subject`: 工单主题
- `content`: 工单内容（TEXT 类型）
- `user_name`: 提交人姓名
- `contact_info`: 联系方式
- `status`: 工单状态（OPEN/IN_PROGRESS/RESOLVED/CLOSED）
- `priority`: 优先级（LOW/NORMAL/HIGH/URGENT）
- `admin_reply`: 管理员回复（TEXT 类型）
- `created_at`: 创建时间（索引）
- `updated_at`: 更新时间
- 索引：`status`、`priority`、`created_at`

### 4.3 关系设计

- **users ↔ bookings**: 一对多关系，一个用户可以有多个订单
- **flights ↔ bookings**: 一对多关系，一个航班可以有多个订单
- **外键约束**: 使用 `ON DELETE RESTRICT` 防止误删关联数据，使用 `ON UPDATE CASCADE` 实现级联更新

## 五、API 接口设计

### 5.1 API 规范

- **Base URL**: `http://localhost:8080`
- **API 前缀**: `/api`
- **请求格式**: JSON (`application/json`)
- **响应格式**: 统一响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

### 5.2 主要接口列表

#### 用户相关接口
- `POST /api/users/register` - 用户注册
- `POST /api/users/login` - 用户登录
- `GET /api/users` - 获取所有用户（管理员）
- `GET /api/users/{id}` - 获取用户详情
- `GET /api/users/stats` - 获取用户统计数据
- `PUT /api/users/{id}` - 更新用户信息
- `PUT /api/users/{id}/password` - 重置用户密码
- `PUT /api/users/{id}/status` - 更新用户状态
- `DELETE /api/users/{id}` - 删除用户

#### 航班相关接口
- `GET /api/flights` - 获取所有航班
- `GET /api/flights/available` - 获取可用航班
- `GET /api/flights/{id}` - 获取航班详情
- `GET /api/flights/number/{flightNumber}` - 根据航班号查询
- `POST /api/flights/search` - 搜索航班
- `POST /api/flights` - 创建航班（管理员）
- `PUT /api/flights/{id}` - 更新航班（管理员）
- `DELETE /api/flights/{id}` - 删除航班（管理员）

#### 预订相关接口
- `POST /api/bookings` - 创建预订

- `GET /api/bookings/{id}` - 获取预订详情
- `GET /api/bookings/user/{userId}` - 获取用户的所有预订
- `GET /api/bookings/flight/{flightId}` - 获取航班的预订列表
- `PUT /api/bookings/{id}/status` - 更新预订状态
- `DELETE /api/bookings/{id}` - 删除预订

#### 工单相关接口
- `POST /api/tickets` - 创建工单
- `GET /api/tickets` - 获取所有工单
- `GET /api/tickets/{id}` - 获取工单详情
- `PUT /api/tickets/{id}/reply` - 回复工单（管理员）
- `PUT /api/tickets/{id}/status` - 更新工单状态
- `DELETE /api/tickets/{id}` - 删除工单

### 5.3 错误处理

系统使用统一的错误响应格式：
```json
{
  "code": 400,
  "message": "错误信息",
  "data": null
}
```

常见 HTTP 状态码：
- `200`: 请求成功
- `400`: 请求参数错误
- `401`: 未授权（需要登录）
- `403`: 禁止访问（权限不足）
- `404`: 资源不存在
- `500`: 服务器内部错误

## 七、部署与运行

### 7.1 环境要求

**后端环境**:
- JDK 17 或更高版本
- Maven 3.6+
- MySQL 8.0+ 或更高版本

**前端环境**:
- Node.js 18+ 或更高版本
- npm 或 yarn 包管理器

### 7.2 数据库配置

1. 创建数据库：
```sql
CREATE DATABASE zjc_airline_booking_db 
    CHARACTER SET utf8mb4 
    COLLATE utf8mb4_unicode_ci;
```

2. 执行建表脚本：
```bash
mysql -u root -p zjc_airline_booking_db < database/schema.sql
```

3. 导入初始数据（可选）：
```bash
mysql -u root -p zjc_airline_booking_db < database/data.sql
```

4. 配置数据库连接（`src/main/resources/application.properties`）：
```properties
spring.datasource.url=jdbc:mysql://localhost:3306/zjc_airline_booking_db?useUnicode=true&characterEncoding=utf8&useSSL=false
spring.datasource.username=root
spring.datasource.password=your_password
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQL8Dialect
```

### 7.3 后端启动

1. 进入项目根目录
2. 编译项目：
```bash
mvn clean install
```

3. 启动应用：
```bash
mvn spring-boot:run
```

或者使用已编译的 JAR：
```bash
java -jar target/zjc-airline-booking-backend-0.1.0-SNAPSHOT.jar
```

4. 后端服务运行在 `http://localhost:8080`

### 7.4 前端启动

1. 进入前端目录：
```bash
cd zjc-airline-booking-frontend
```

2. 安装依赖：
```bash
npm install
```

3. 启动开发服务器：
```bash
npm run dev
```

4. 前端应用运行在 `http://localhost:3000`

### 7.5 生产环境部署

**后端部署**:
- 使用 `mvn package` 打包成 JAR 文件
- 部署到服务器，使用 `java -jar` 运行
- 建议使用 Nginx 作为反向代理
- 配置生产环境的数据库连接

**前端部署**:
- 使用 `npm run build` 构建生产版本
- 使用 `npm start` 启动生产服务器
- 或部署到 Vercel、Netlify 等平台
- 配置环境变量指向生产后端 API

## 八、开发规范与最佳实践

### 8.1 代码规范

**后端**:
- 遵循 Java 命名规范（驼峰命名）
- Controller 层只负责参数校验和调用 Service
- Service 层处理业务逻辑
- Repository 层只负责数据访问
- 使用 DTO 进行数据传输，避免直接暴露 Entity

**前端**:
- 使用 TypeScript 进行类型约束
- 组件采用函数式组件和 Hooks
- 遵循 React 最佳实践（单一职责、组件复用）
- 使用 ESLint 进行代码检查

### 8.2 安全实践

- 密码使用 BCrypt 加密，永不存储明文
- API 接口进行参数验证
- 前端进行输入验证和 XSS 防护
- 使用 HTTPS 传输敏感数据（生产环境）
- 定期更新依赖包，修复安全漏洞

### 8.3 性能优化

- 数据库查询使用索引优化
- 前端使用 Next.js 自动代码分割
- 图片使用 Next.js Image 组件优化
- API 响应使用分页减少数据传输
- 使用缓存策略减少重复请求

### 8.4 测试

- 单元测试：使用 JUnit 测试后端业务逻辑
- 集成测试：测试 API 接口的完整流程
- 前端测试：使用 React Testing Library 测试组件
- API 测试：使用 Postman Collection 进行接口测试
