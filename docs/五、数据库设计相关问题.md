## 1. 请描述系统的数据库表结构，包括主要表和它们之间的关系？

### 1.1 核心表结构

#### 1.1.1 users（用户表）

**字段：**
- `id`：用户ID（主键，自增）
- `username`：用户名（唯一）

#### 1.1.2 flights（航班表）

- `destination`：目的地

- `arrival_time`：到达时间
- `remarks`：备注
- `created_at`：创建时间
- `updated_at`：更新时间

**索引：**
- `idx_flight_number`：航班号索引
- `idx_origin`：出发地索引
- `idx_destination`：目的地索引
- `idx_departure_time`：出发时间索引

#### 1.1.3 flight_seats（航班舱位表 - 旧系统）

**字段：**
- `id`：舱位ID（主键，自增）
- `flight_id`：航班ID（外键）
- `cabin_class`：舱位等级（ECONOMY/BUSINESS/FIRST）

- `remaining_seats`：剩余座位数

- `updated_at`：更新时间

**外键：**
- `flight_id` → `flights.id`（ON DELETE CASCADE）

#### 1.1.4 flight_prices（航班价格表 - 新系统）

**字段：**
- `id`：价格ID（主键，自增）
- `flight_id`：航班ID（外键）
- `cabin_class`：舱位等级
- `base_price`：基础价格

**外键：**
- `flight_id` → `flights.id`

#### 1.1.5 seats（座位表 - 新系统）

**字段：**
- `id`：座位ID（主键，自增）
- `flight_id`：航班ID（外键）
- `cabin_class`：舱位等级
- `seat_number`：座位号（如：E1A）
- `seat_row`：排号
- `seat_column`：列号
- `is_available`：是否可用

**外键：**
- `flight_id` → `flights.id`

#### 1.1.6 bookings（预订表）

**字段：**
- `id`：预订ID（主键，自增）
- `flight_id`：航班ID（外键）
- `user_id`：用户ID（外键）
- `cabin_class`：舱位等级
- `ticket_count`：票数
- `total_amount`：总金额
- `status`：状态
- `order_no`：订单号（唯一）
- `payment_due_at`：支付截止时间
- `urgent_surcharge_rate`：紧急加价比例

**索引：**
- `idx_user_id`：用户ID索引
- `idx_flight_id`：航班ID索引
- `idx_status`：状态索引
- `idx_created_at`：创建时间索引

#### 1.1.7 booking_seats（预订座位表）

**字段：**
- `id`：ID（主键，自增）

## 2. 为什么使用外键约束？有什么优缺点？

### 2.1 外键约束的作用

**数据完整性：**
- 确保引用完整性（Referential Integrity）
- 防止孤立记录（Orphaned Records）
- 自动维护数据一致性

**示例：**
```sql
FOREIGN KEY (flight_id) REFERENCES flights(id) ON DELETE RESTRICT
```
- 删除航班时，如果有订单关联，则拒绝删除（RESTRICT）
- 防止订单指向不存在的航班

### 2.2 外键约束的优点

**1. 数据完整性保证**
- ✅ 自动检查引用完整性
- ✅ 防止非法数据插入
- ✅ 级联操作（CASCADE、SET NULL）

**2. 代码简化**
- ✅ 减少应用层验证代码
- ✅ 数据库自动维护关系

**3. 文档化**
- ✅ 表关系一目了然
- ✅ 便于理解和维护

### 2.3 外键约束的缺点

**1. 性能影响**
- ❌ 插入/更新时需要检查外键
- ❌ 删除时需要检查引用
- ❌ 可能影响并发性能

**2. 灵活性降低**
- ❌ 不能随意删除被引用的记录
- ❌ 需要先处理关联数据

**3. 分布式系统**
- ❌ 跨数据库外键不支持
- ❌ 微服务架构中难以使用

### 2.4 项目中的使用

**当前策略：**
- ✅ 使用外键约束保证数据完整性
- ✅ ON DELETE RESTRICT：防止误删关键数据
- ✅ ON DELETE CASCADE：级联删除关联数据

**示例：**
```sql
-- 订单关联航班（不能删除有关联订单的航班）
FOREIGN KEY (flight_id) REFERENCES flights(id) ON DELETE RESTRICT

-- 舱位关联航班（删除航班时级联删除舱位）
FOREIGN KEY (flight_id) REFERENCES flights(id) ON DELETE CASCADE
```

## 3. 如何设计索引以提高查询性能？

### 3.1 索引设计原则

**1. 为常用查询字段创建索引**
- 主键自动创建索引
- 外键创建索引
- 唯一字段创建索引
- 频繁查询的字段创建索引

**2. 联合索引优化多条件查询**
```sql
-- 出发地+目的地联合查询
INDEX idx_route (origin, destination)
```

**3. 避免过度索引**
- 索引会占用存储空间
- 插入/更新时需要维护索引
- 过多的索引可能降低写入性能

### 3.2 项目中的索引设计

**users 表：**
```sql
INDEX idx_username (username)        -- 登录查询
INDEX idx_id_number (id_number)      -- 注册验证
INDEX idx_phone (phone)              -- 注册验证
```

**flights 表：**
```sql
INDEX idx_flight_number (flight_number)    -- 航班号查询
INDEX idx_origin (origin)                  -- 出发地搜索
INDEX idx_destination (destination)        -- 目的地搜索
INDEX idx_departure_time (departure_time)  -- 时间排序
INDEX idx_route (origin, destination)      -- 联合查询
```

**bookings 表：**
```sql
INDEX idx_user_id (user_id)          -- 用户订单查询
INDEX idx_flight_id (flight_id)      -- 航班订单统计
INDEX idx_status (status)            -- 状态筛选
INDEX idx_created_at (created_at)    -- 时间排序
```

### 3.3 索引类型选择

**B-Tree 索引（默认）：**
- 适用于等值查询、范围查询
- 大多数场景使用

**Hash 索引：**
- 适用于等值查询
- 不适合范围查询

**全文索引：**
- 适用于文本搜索
- MySQL 5.6+ 支持 InnoDB 全文索引

## 4. 如何处理数据的一致性和完整性？

### 4.1 数据库约束

**主键约束：**
```sql
PRIMARY KEY (id)
```
- 保证唯一性
- 非空

**唯一约束：**
```sql
UNIQUE KEY UK_order_no (order_no)
```
- 保证字段唯一性

**非空约束：**
```sql
NOT NULL
```
- 保证字段必须有值

**外键约束：**
```sql
FOREIGN KEY (flight_id) REFERENCES flights(id)
```
- 保证引用完整性

### 4.2 事务管理

**ACID 特性：**
- **原子性（Atomicity）：** 事务要么全部成功，要么全部回滚
- **一致性（Consistency）：** 事务前后数据保持一致
- **隔离性（Isolation）：** 事务之间互不影响
- **持久性（Durability）：** 提交后数据永久保存

**Spring 事务：**
```java
@Transactional
public Booking createBooking(BookingRequest request) {
    // 所有操作在同一事务中
    // 出现异常自动回滚
}
```

### 4.3 应用层验证

**业务规则验证：**
```java
// 检查座位数量
if (remainingSeats < ticketCount) {
    throw new RuntimeException("座位不足");
}

// 检查钱包余额
if (walletBalance.compareTo(totalAmount) < 0) {
    throw new RuntimeException("余额不足");
}
```

## 5. 如何实现数据库的备份和恢复？

### 5.1 备份策略

**全量备份：**
```bash
# 备份整个数据库
mysqldump -u root -p zjc_airline_booking_db > backup_$(date +%Y%m%d).sql
```

**增量备份：**
- 使用 MySQL 二进制日志（binlog）
- 定期备份 binlog 文件

**备份频率：**
- 全量备份：每天一次
- 增量备份：每小时一次

### 5.2 恢复策略

**全量恢复：**
```bash
# 恢复数据库
mysql -u root -p zjc_airline_booking_db < backup_20251120.sql
```

**增量恢复：**
```bash
# 恢复 binlog
mysqlbinlog binlog.000001 | mysql -u root -p zjc_airline_booking_db
```

### 5.3 备份工具

**mysqldump：**
- MySQL 官方工具
- 适合小型数据库

**MySQL Enterprise Backup：**
- MySQL 官方商业工具
- 支持热备份

**第三方工具：**
- Percona XtraBackup
- mydumper
