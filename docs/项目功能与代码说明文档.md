## 一、项目总体介绍

本项目是一个以 **航空机票预订场景** 为背景的完整 Web 系统，包含：

- **后端服务（`zjc-airline-booking-backend`）**：
  - 基于 Spring Boot 3.3.0 开发
  - 负责业务逻辑处理、数据库访问、价格/规则计算、订单与座位的并发控制等
  - 通过 RESTful API 对外提供服务，前端通过 HTTP/JSON 调用

- **前端应用（`zjc-airline-booking-frontend`）**：
  - 基于 Next.js 14 + TypeScript 实现
  - 提供航班搜索、机票预订、订单管理、用户中心、管理员后台等界面
  - 通过 `src/lib/*.ts` 中封装的 API 客户端访问后端

- **数据库层（MySQL）**：
  - 通过 `spring-boot-starter-data-jpa` 访问
  - 使用实体类 + JPA 仓库管理航班、用户、订单、座位等核心表

- **文档与架构说明（`docs/` 目录）**：
  - 存放多个专题文档（功能实现、架构、安全、性能、数据库设计等）
  - 当前这个文档主要从“**功能 → 对应代码**”的视角梳理系统

后端的基础配置定义在：

- `pom.xml`
- `src/main/resources/application.properties`

前端的基础配置与依赖定义在：

- `zjc-airline-booking-frontend/package.json`
- `zjc-airline-booking-frontend/next.config.js`
- `zjc-airline-booking-frontend/src/app/layout.tsx`

下面将从各个核心功能出发，分别说明：

1. 功能在系统中的业务含义
2. 前端页面/组件对应位置
3. 调用的前端 API 封装代码

### 3.2 前端数据结构与映射（`flights.ts`）

文件路径：`zjc-airline-booking-frontend/src/lib/flights.ts`

核心类型：

- `FlightQuery`：前端搜索条件
- `Flight`：前端统一航班模型
- `FlightSeat`：前端舱位信息结构
- `FlightPayload` & `FlightSeatPayload`：创建/更新航班时的请求体

例如：

- `FlightQuery` 包含 `date`, `from`, `to` 三个可选字段
- `Flight` 中的主要字段：
  - `flightNumber`, `airline`, `aircraftType`, `from`, `to`, `departTime`, `arriveTime`, `duration`, `price`, `seatsLeft`, `seats`, `status`, `remarks`

**核心映射函数 `mapFlight`：**

- 输入：后端返回的 `FlightApiEntity`（字段如 `flightNumber`, `origin`, `destination`, `departureTime`, `arrivalTime`, `seats`, `planeType` 等）
- 输出：标准化后的 `Flight`
- 关键逻辑：
  - 通过舱位数组 `data.seats`：
    - 计算总剩余座位数 `totalRemainingSeats`
    - 计算最低票价 `minPrice`
  - 映射为前端可用的 `seats: FlightSeat[]` 列表
  - 将状态字段映射为统一的 `ON_TIME` / `DELAYED` / `CANCELLED` 等

**响应归一化 `normalizeResponse`：**

- 考虑到不同后端接口可能返回：
  - `data.data` / `data.records` / `data.list` / `data.rows` / `data.content` 等结构
- 通过 `candidateArrays` 数组依次尝试这些字段，找到真正的数组数据

> 这一层做了“**弱耦合兼容**”，可以在后端调整分页或封装格式时，尽量不影响前端的大部分调用代码。

### 3.3 航班列表 API 映射

在 `flightApi` 对象中，主要有以下方法：

- `listAll(options?: { includePast?: boolean })`：
  - 请求：`GET /api/flights`
  - 参数：`includePast`（是否包含历史航班）
  - 使用：在管理后台或高级页面中展示全部航班

- `listAvailable()`：
  - 请求：`GET /api/flights/available`
  - 返回所有可售航班

- `recommendations(options?: { limit?: number; includePast?: boolean })`：
  - 请求：`GET /api/flights/recommendations`
  - 用于首页推荐区块

- `search(query: FlightQuery)`：
  - 请求：`POST /api/flights/search`
  - 请求体字段：`origin`, `destination`, `departureDate`
  - 特点：在请求前对 `query` 中的 `from` / `to` / `date` 做格式清洗

### 3.4 后端航班查询实现映射

依据 `docs/二、功能实现相关问题.md` 中的完整示例，航班接口核心位置：

- `FlightController`：
  - `@GetMapping("/api/flights")` → 对应前端的 `flightApi.listAll`
  - `@PostMapping("/api/flights/search")` → 对应前端的 `flightApi.search`

- `FlightService` 中的方法：
  - `getAllFlights(includePast)`：
    - 调用 `flightRepository.findAll()` 或 `findAvailableFlights()`，并根据 `includePast` 过滤
  - `searchFlights(FlightSearchRequest request, boolean includePast)`：
    - 使用 `normalizeKeyword` 对出发地和目的地关键字进行处理
    - 根据条件组合调用：
      - `findByOriginContainingIgnoreCaseAndDestinationContainingIgnoreCase(...)`
      - `findByOriginContainingIgnoreCase(...)`
      - `findByDestinationContainingIgnoreCase(...)`
      - 或 `findAvailableFlights()`

- `FlightRepository`：
  - 典型方法如下（声明在接口中）：
    - `List<Flight> findByOriginContainingIgnoreCase(String origin);`
    - `List<Flight> findByDestinationContainingIgnoreCase(String destination);`
    - `List<Flight> findByOriginContainingIgnoreCaseAndDestinationContainingIgnoreCase(String origin, String destination);`
    - `@Query` + `WHERE departure_time >= NOW()` 用于 `findAvailableFlights()`

这样前后端建立了稳定的映射关系：

- **前端组件**：城市选择、日期选择 → 组装为 `FlightQuery`
- **前端 API 层**：`flightApi.search` → 提交至 `/api/flights/search`
- **后端 Controller**：`FlightController.searchFlights` → 调用 `FlightService`
- **后端 Service**：`FlightService.searchFlights` → 基于 JPA Repository 组合查询
- **后端 Repository**：封装实际数据库查询

## 四、用户注册与登录功能

### 4.1 功能说明

**注册**：

- 用户通过注册页面填写：用户名、密码、确认密码、姓名、身份证号、联系电话、地址等
- 系统验证：
  - 用户名唯一
  - 身份证号唯一
  - 手机号唯一
- 密码在后端使用 BCrypt 加密后存储
- 初始角色为 `USER`
- 初始钱包余额默认为 200 元

**登录**：

- 用户输入用户名和密码
- 系统验证用户名与密码是否匹配
- 登录成功后返回用户信息（不含明文密码），前端保存到本地，用于后续请求

### 4.2 前端页面与代码位置

- 注册页：`src/app/register/page.tsx`
- 登录页：`src/app/login/page.tsx`
- 前端认证工具：`src/lib/auth.ts`
- 用户相关 API 封装：`src/lib/users.ts`（通常包含 `login`, `register`, `getProfile` 等接口）

其中 `auth.ts` 中核心工具：

- `saveAuth(userInfo: UserInfo)`：将用户信息和 Token 保存到 `localStorage` 中
- `getAuth(): UserInfo | null`：从 `localStorage` 中读取
- `clearAuth()`：清除本地认证信息
- `isAuthenticated()`：判断当前是否已登录（主要依据是否存在 Token）
- `getToken()`：获取 Token，用于在 `api.ts` 中的请求头中自动附加

前端登录/注册页面的调用流程一般为：

1. 收集表单数据
2. 调用 `usersApi.login` 或 `usersApi.register`（定义在 `users.ts`）
3. 成功后调用 `saveAuth`，并根据角色跳转不同页面（普通用户跳转首页，管理员跳转后台）

### 4.3 后端用户模块代码

- `UserController`：
  - `@PostMapping("/api/users/register")` → 注册接口
  - `@PostMapping("/api/users/login")` → 登录接口

- `UserService`：
  - `register(RegisterRequest request)`：
    - 调用 `userRepository.findByUsername` / `findByIdNumber` / `findByPhone` 做唯一性检查
    - 使用 `PasswordEncoder`（BCrypt）加密密码
    - 初始化角色和钱包余额
    - 保存至数据库
  - `login(LoginRequest request)`：
    - 根据用户名查询用户
    - `passwordEncoder.matches(plain, encoded)` 验证密码
    - 验证不通过统一抛出“用户名或密码错误”

- `SecurityConfig`：
  - 暂时将大部分接口 `permitAll()`，后续可升级为基于角色/Token 的细粒度控制

**密码安全对应代码：**

- `SecurityConfig.passwordEncoder()`：
  - 返回 `BCryptPasswordEncoder` 实例
- `UserService.register()`：
  - `user.setPassword(passwordEncoder.encode(request.getPassword()));`

## 五、机票预订、座位选择与价格计算

### 5.1 前端预订流程与代码

**用户操作路径：**

1. 在航班列表（如首页 `page.tsx`）中选择某个航班并点击“预订”
2. 在预订页面中：
   - 选择舱位（经济舱/商务舱/头等舱）
   - 选择订票数量
   - 填写乘客信息
   - 选择是否“稍后支付”或立即支付
   - 选择座位列（如选择 `A/B/C/D` 列）
3. 提交预订请求

**前端代码对应：**

- 页面：`src/app/bookings/new/page.tsx` 或航班详情页中的弹窗/面板
- API：`src/lib/bookings.ts` 中的：
  - `bookingApi.create(payload: BookingRequest)`
  - `seatApi.getAvailableSeatColumns(flightId, cabinClass)`
  - `seatApi.selectSeats(request: SeatSelectionRequest)`

`BookingRequest` 结构：

- 字段包括：
  - `flightId: number`
  - `userId: number`
  - `cabinClass: string`
  - `ticketCount: number`
  - `payLater?: boolean`
  - `selectedSeatColumns?: string[]`
  - `passengers?: PassengerPayload[]`

这与后端 `BookingRequest` DTO 一一对应，后端在 `BookingService.createBooking` 中解析并执行业务逻辑。

### 5.2 后端预订核心代码映射

- `BookingController`：
  - `@PostMapping("/api/bookings")` → 创建预订
  - `@GetMapping("/api/bookings/user/{userId}")` → 查询用户订单
  - `@PostMapping("/api/bookings/{id}/pay")` → 支付订单
  - `@PostMapping("/api/bookings/{id}/cancel")` → 取消订单
  - `@PostMapping("/api/bookings/{id}/refund-request")` → 申请退票
  - `@PostMapping("/api/bookings/{id}/refund-review")` → 管理员审核退票
  - `@GetMapping("/api/bookings/{id}/change/check")` → 检查是否可改签
  - `@PostMapping("/api/bookings/{id}/change/price")` → 计算改签价格
  - `@PostMapping("/api/bookings/{id}/change")` → 执行改签

- `BookingService`：
  - `createBooking(BookingRequest request)`：
    - 验证乘客身份证与手机号唯一性
    - 校验所选舱位是否有足够剩余座位
    - 根据舱位价格与紧急加价规则计算订单金额
    - 根据 `payLater` 决定是否立即扣费
    - 调用座位服务分配具体座位号
    - 保存订单与关联的 `BookingSeat` 记录

- `SeatService`：
  - `getAvailableSeatColumns(flightId, cabinClass)`：
    - 返回当前舱位可用座位列/列余量
  - `selectSeats(SeatSelectionRequest request)`：
    - 结合 `flightId`、`cabinClass`、`ticketCount`、`selectedColumns` 进行座位分配
    - 返回 `assignedSeats` 等信息，供 `BookingService` 关联保存

### 5.3 前端座位选择 API 对应后端接口

在 `bookings.ts` 中：

- `seatApi.getAvailableSeatColumns(flightId, cabinClass)`：
  - 请求：`GET /api/seats/available?flightId=...&cabinClass=...`
  - 返回：`SeatSelectionResponse`

- `seatApi.selectSeats(request: SeatSelectionRequest)`：
  - 请求：`POST /api/seats/select`
  - 请求体：`{ flightId, cabinClass, ticketCount, selectedColumns }`

后端对应：

- `SeatController.getAvailableSeats(...)` / `SeatController.getAvailableSeatColumns(...)`
- `SeatController.selectSeats(...)`
- `SeatService` 内实际的分配与校验逻辑

### 5.4 价格与规则实现位置

**后端规则实现（在 `BookingService` 或相关规则类中）：**

- 紧急加价：
  - 如果在起飞前 12 小时内预订，基础价格上增加 10% 紧急附加费
  - 关键字段：`URGENT_SURCHARGE_RATE`

- 退票手续费：
  - 起飞前 12 小时内退票，收取 1% 手续费
  - 相关方法：`calculateRefundAmount(Booking booking)`

**前端规则提示（`booking-rules.ts`）：**

- 用于在 UI 上给出中文提示和事前说明
- 避免用户产生“预期外”的费用变动

## 六、订单管理、支付、取消、退票与改签

### 6.1 前端订单管理页面与代码

主要页面路径：

- `src/app/bookings/page.tsx`：订单列表页
- `src/app/bookings/[id]/page.tsx`：订单详情页

主要 API 封装：

- `bookingApi.list(query?: BookingListQuery)`：
  - 请求：`GET /api/bookings` 或 `GET /api/bookings/user/{userId}`
  - 响应归一化后映射为 `BookingItem[]`

- `bookingApi.pay(bookingId, userId)`：
  - 请求：`POST /api/bookings/{id}/pay`

- `bookingApi.cancel(bookingId, userId)`：
  - 请求：`POST /api/bookings/{id}/cancel`

- `bookingApi.requestRefund(bookingId, { userId, reason })`
- `bookingApi.reviewRefund(bookingId, { approve, reason })`
- `bookingApi.checkCanChange(bookingId, userId)`
- `bookingApi.calculateChangePrice(bookingId, userId, request)`
- `bookingApi.changeBooking(bookingId, userId, request)`

这些方法在 `bookings.ts` 中全部定义，对应的后端接口统一由 `BookingController` 管理。

### 6.2 后端订单状态与流程实现

**订单状态字段（枚举字符串）：**

- `CREATED`：已创建未支付
- `PAID`：已支付
- `REFUND_REVIEW`：退票待审核
- `REFUND_REJECTED`：退票被拒
- `CANCELED`：已取消

对应实体：`Booking`（在 `com/example/demo/booking` 包下）

**关键流程：**

- 创建订单：`BookingService.createBooking`（见第 5 节）
- 支付订单：`BookingService.payBooking(bookingId, userId)`
- 取消订单：`BookingService.cancelBooking(bookingId, userId)`
- 申请退票：`BookingService.requestRefund(bookingId, reason)`
- 管理员审核退票：`BookingService.approveRefund(...)` / `rejectRefund(...)`
- 改签：`BookingService.changeBooking(bookingId, ChangeBookingRequest request)`

**超时未支付自动取消：**

- 在 `BookingService` 或独立 `ScheduledTask` 类中：
  - `@Scheduled(fixedRate = 60000)`
  - 定期查询 `status = CREATED` 且 `paymentDueAt < now` 的订单
  - 设置状态为 `CANCELED` 并释放座位
